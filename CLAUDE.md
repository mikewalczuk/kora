# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Rules

- After every code change, run `make test` from `backend/` and fix any failures before considering the task done.
- Use the TodoWrite tool to track multi-step tasks (e.g. edit spec → regenerate types → update handler → run tests). Mark each step done as you complete it.

## How to work on this project

### 1. Design the API first — edit `openapi/openapi.yaml`

The OpenAPI spec is the single source of truth. Every endpoint, request body, and response schema lives there. Do not write handler signatures or types by hand — define them in the spec and generate.

### 2. Write SQL queries — edit `internal/<feature>/queries.sql`

Add raw SQL queries annotated with sqlc directives:

```sql
-- name: GetUserByUsername :one
SELECT * FROM users WHERE username = $1;
```

Migrations (table definitions) go in `migrations/XXX_name.sql` using goose format.

### 3. Generate everything

```bash
make gen   # runs oapi-codegen + sqlc + go mod tidy
```

This produces:

- `internal/api/api.gen.go` — HTTP types, router wiring, strict-server interface (**do not edit**)
- `internal/database/` — type-safe query methods (**do not edit**)

### 4. Implement the feature

Create a feature folder under `internal/<feature>/` with:

```
internal/<feature>/
  handler.go    ← implements StrictServerInterface methods
  service.go    ← business logic; has Service struct with DB field
  queries.sql   ← SQL source for sqlc
```

Handlers receive already-parsed typed request objects and return typed response objects — no manual JSON decoding needed.

### 5. Run dev server

```bash
make dev   # gen → migrate-up → air (hot reload)
```

Requires a `.env` file in `backend/` with at least `DATABASE_URL=postgres://...`.

### 6. Migrations

```bash
make migrate-up    # apply pending migrations
make migrate-down  # roll back one migration
```

Migration files live in `backend/migrations/`. Name them `YYYYMMDDHHMMSS_description.sql`.

## Frontend code generation (Orval)

The frontend TypeScript hooks are generated from the same `api-spec/openapi.yaml` using **Orval**. Run from `frontend/`:

```bash
npm run gen:api   # regenerates src/lib/api/generated/ from the OpenAPI spec
```

Output is split by tag (e.g. `generated/auth/`, `generated/notes/`). Schemas land in `generated/koraAPI.schemas.ts`.

### ⚠️ customFetch / Orval type mismatch

Orval generates response types shaped as `{ data: Schema, status: N, headers: Headers }`, but the project's `customFetch` (`src/lib/api/mutator.ts`) returns the **raw JSON body** directly. This means:

- At runtime `useXxx().data` is `Schema`, not `{ data: Schema, status: 200, ... }`
- Accessing `data.data.items` or `data.status` will be `undefined` at runtime
- To fix, cast in a feature wrapper: `data as unknown as Schema`

**Pattern**: each feature has `src/features/<name>/api.ts` that wraps generated hooks and hides the cast:

```ts
// features/notes/api.ts
export function useNotesList(params?: ListNotesParams) {
  const { data, isLoading, isError } = _useListNotes(params);
  const response = data as unknown as ListNotesResponse | undefined;
  return { notes: response?.items ?? [], isLoading, isError };
}
```

Components import from `features/<name>/api.ts`, never from `lib/api/generated/` directly.

### Frontend structure

```
frontend/
  orval.config.ts              ← Orval config (input: openapi.yaml, output: tags-split)
  src/
    lib/api/
      mutator.ts               ← customFetch — returns raw JSON body, throws on non-2xx
      generated/               ← DO NOT EDIT — generated by Orval
        koraAPI.schemas.ts     ← all schema types
        <tag>/                 ← hooks per tag (auth/, notes/, …)
    features/<name>/
      api.ts                   ← re-exports / wraps generated hooks; hides type cast
      *.tsx                    ← components
    routes/                    ← TanStack Router file-based pages
```

## Backend: combining multiple feature handlers

`api.StrictServerInterface` must be satisfied by a single type. When each feature has its own `Handler` struct, **do not embed** (two types named `Handler` create an ambiguous field). Instead, use a `server` struct in `cmd/api/main.go` with named fields and one-liner delegation:

```go
type server struct {
    auth  *auth.Handler
    notes *notes.Handler
}

var _ api.StrictServerInterface = (*server)(nil)

func (s *server) Login(ctx context.Context, req api.LoginRequestObject) (api.LoginResponseObject, error) {
    return s.auth.Login(ctx, req)
}
// … one line per method
```

The compile-time interface check (`var _ api.StrictServerInterface = (*server)(nil)`) belongs here, not inside individual feature packages.

## Vibe-coding workflow (AI-assisted development)

When starting a new feature:

1. Define the endpoints in `api-spec/openapi.yaml` (schemas, request/response bodies, status codes)
2. Run `make gen` to generate the interface
3. Create skeleton files — handler with empty method stubs that compile, service with the struct — **no logic yet**
4. Clarify requirements, edge cases, error handling
5. Implement logic only after the skeleton compiles and requirements are clear

This prevents wasted work: the generated interface tells you exactly what signatures to implement before writing a single line of business logic.

## Known caveats & gotchas

### ⚠️ Always bundle before generating

`make gen` in `backend/` uses `openapi.bundled.yaml` — a pre-flattened snapshot. It does **not** re-bundle automatically. If you edit any file under `openapi/`, you must re-bundle first or both oapi-codegen and Orval will see stale types:

```bash
cd frontend && npm run bundle:api   # rebuild openapi.bundled.yaml
cd ../backend && make gen           # then regen Go types + sqlc
npm run gen:api                     # regen TS hooks (also reads bundled file)
```

Full cross-stack regen sequence: `bundle:api` → `make gen` → `gen:api`.

### ⚠️ `uuid.UUID` serializes as a byte array

`uuid.UUID` in Go is `[16]byte`. Passing it directly to `json.Marshal` or an SSE broadcast produces `[111, 205, 34, …]`, not a UUID string. Always call `.String()`:

```go
// Wrong — produces a byte array in JSON
"practiceId": practiceID

// Right
"practiceId": uuid.UUID(practiceID).String()   // if practiceID is [16]byte
"practiceId": practiceID.String()              // if already uuid.UUID
```

### ⚠️ oapi-codegen may skip a 409 if the response body is reused

Referencing `$ref: '../components/responses/Conflict.yaml'` (which has a JSON body) for a second endpoint can silently drop the 409 type. For endpoints that need a no-body 409, define it inline:

```yaml
"409":
  description: Already exists
```

### Transactions with sqlc

`database.Queries` has a `WithTx` helper. Store `*pgxpool.Pool` on the service alongside `*database.Queries`:

```go
tx, err := s.Pool.Begin(ctx)
defer tx.Rollback(ctx)
q := s.DB.WithTx(tx)
// ... all DB ops via q ...
tx.Commit(ctx)
```

### sqlc queries are per-feature but share one output package

All `internal/*/queries.sql` files compile into the single `internal/database` package. A query in `practices/queries.sql` that reads from the `notes` table is fine — sqlc sees all tables from the migrations schema. Add cross-table queries to whichever feature's `queries.sql` needs them.

### AI port/adapter pattern

`internal/ai` is the port (interface + value types only). `mock.go` and `openai.go` are adapters. The interface signature:

```go
Generate(ctx, noteTitle, noteContent string, existingConcepts []Concept) (GenerateResult, error)
```

`existingConcepts` carries IDs so the AI can reference them in `ObsoletedConceptIDs`. The mock ignores all inputs. To swap to a real provider, change one line in `main.go`.

## Architecture overview

```
openapi/openapi.yaml           ← source of truth for the HTTP API (split into paths/ + components/)
backend/
  cmd/api/main.go              ← wires router, middleware, DB pool, handlers
  internal/
    api/api.gen.go             ← DO NOT EDIT — generated from openapi.yaml
    database/                  ← DO NOT EDIT — generated by sqlc
    config/                    ← env-based config (PORT, DATABASE_URL, LOG_FORMAT)
    testutils/                 ← shared test helpers (Postgres container)
    auth/                      ← auth feature (handler, service, queries)
  migrations/                  ← goose SQL migrations
  tools.go                     ← pins CLI tools (goose) in go.mod; build tag "tools"
```

### Request lifecycle

`chi router` → `OapiRequestValidator` (validates against spec) → `StrictHandler` (decodes body) → `Handler` method (typed request/response objects)

Handlers never see raw JSON — the middleware pipeline handles validation and decoding.

### Testing

Integration tests use real Postgres via testcontainers. Pattern:

- `TestMain` starts one shared container for the package; `TRUNCATE` between tests for isolation
- `testutils.NewPostgresPool(ctx, schema)` — for `TestMain` (returns teardown func)
- `testutils.StartPostgres(ctx, t, schema)` — for per-test use (registers `t.Cleanup`)
